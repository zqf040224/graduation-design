# 知识库构建指南

## 一、资料准备

### 1.1 目录结构

在电脑上创建一个文件夹，例如 `单位公文资料库`，按以下结构组织资料：

```
单位公文资料库/
├── 公文规范/
│   ├── 国家标准公文格式.docx
│   ├── 单位公文格式细则.docx
│   └── 字体字号规范.docx
│
├── 通知模板/
│   ├── 会议通知模板.docx
│   ├── 放假通知模板.docx
│   └── 人事任免通知模板.docx
│
├── 请示模板/
│   ├── 经费请示模板.docx
│   └── 活动请示模板.docx
│
├── 报告模板/
│   ├── 工作报告模板.docx
│   └── 调研报告模板.docx
│
├── 优秀范文/
│   ├── 2024 年优秀公文 10 篇/
│   └── 经典公文案例.docx
│
└── 常用短语/
    ├── 开头用语.txt
    ├── 结尾用语.txt
    └── 过渡用语.txt
```

### 1.2 资料要求

**公文规范文档**应包含：
- ✅ 标题格式（字体、字号、是否居中）
- ✅ 正文格式（字体、字号、行距、缩进）
- ✅ 落款格式（部门、日期、位置）
- ✅ 文号格式（如"深人深院〔2025〕1 号"）
- ✅ 版式要求（页边距、页码位置）

**模板文档**应包含：
- ✅ 完整的公文结构
- ✅ 占位符标注（如"[会议名称]"、"[时间]"）
- ✅ 固定用语（如"经研究决定"、"特此通知"）

**优秀范文**应包含：
- ✅ 真实的历史公文（脱敏后）
- ✅ 涵盖不同类型（通知、请示、报告等）
- ✅ 格式规范、内容完整


```bash

cd /Users/qfen9/Documents/code/Agent
```

## 三、知识库内容示例

### 3.1 公文规范会被这样解析

**原始文档**（国家标准公文格式.docx）：
```
一、标题
标题使用黑体二号字，居中排列。

二、正文
正文使用仿宋_GB2312 三号字，行距 28 磅，首行缩进 2 字符。

三、落款
落款包括发文机关和日期，居右对齐。
```

**解析后存入知识库**：
```json
{
  "content": "标题使用黑体二号字，居中排列。",
  "metadata": {
    "source": "公文规范/国家标准公文格式.docx",
    "category": "公文规范",
    "section": "标题格式"
  }
}
```

### 3.2 模板会被这样解析

**原始文档**（会议通知模板.docx）：
```
关于召开 [会议名称] 的通知

各部门：
经研究决定，定于[时间] 在 [地点] 召开 [会议名称]。现将有关事项通知如下：

一、会议时间：[具体时间]
二、会议地点：[具体地点]
三、参会人员：[参会范围]
四、会议内容：[主要议程]

请各单位提前做好安排，准时参会。

特此通知。

[发文部门]
[日期]
```

**解析后存入知识库**：
```json
{
  "content": "关于召开 [会议名称] 的通知\n\n各部门：\n经研究决定，定于 [时间] 在 [地点] 召开 [会议名称]...",
  "metadata": {
    "source": "通知模板/会议通知模板.docx",
    "category": "通知模板",
    "template_type": "会议通知"
  }
}
```

## 四、Agent 如何使用知识库

### 4.1 用户生成公文的流程

1. **用户输入需求**：
   ```
   帮我写一份会议通知，下周三下午 2 点在 302 会议室召开
   "2025 年度科研工作总结会"，参会人员是各部门负责人。
   ```

2. **Agent 检索知识库**：
   - 检索"通知模板" → 找到会议通知模板
   - 检索"公文格式" → 找到格式规范
   - 检索"常用短语" → 找到"经研究决定"、"特此通知"等

3. **Agent 生成公文**：
   ```
   关于召开 2025 年度科研工作总结会的通知

   各部门：
   经研究决定，定于 2025 年 1 月 15 日（星期三）下午 2 点在 302 会议室
   召开 2025 年度科研工作总结会。现将有关事项通知如下：

   一、会议时间：2025 年 1 月 15 日下午 2:00
   二、会议地点：302 会议室
   三、参会人员：各部门负责人
   四、会议内容：总结 2025 年度科研工作，部署 2026 年计划

   请各单位提前做好安排，准时参会。

   特此通知。

   智能知识库平台
   2025 年 1 月 8 日
   ```

4. **格式自动套用**：
   - 标题：黑体二号，居中
   - 正文：仿宋_GB2312 三号，28 磅行距
   - 落款：居右对齐

## 六、最佳实践

### 6.1 资料质量建议

✅ **推荐**：
- 使用 Word 格式（.docx），便于解析
- 文档结构清晰，有明确标题层级
- 模板包含完整结构和占位符
- 范文涵盖不同类型和场景

❌ **避免**：
- 扫描版 PDF（无法提取文字）
- 格式混乱、结构不清的文档
- 包含敏感信息的文档（需脱敏）
- 过时的公文规范

### 6.2 知识库规模建议

| 知识库类型 | 文件数 | 片段数 | 大小 | 适用场景 |
|-----------|--------|--------|------|---------|
| **精简版** | 20-30 | 150-250 | 5-10MB | 快速启动、测试 |
| **标准版** | 40-60 | 300-500 | 15-25MB | 日常使用（推荐） |
| **完整版** | 80-120 | 600-1000 | 40-60MB | 功能完备 |

### 6.3 测试建议

构建完成后，建议测试：

```python
# 测试检索准确性
from langchain.vectorstores import FAISS
from langchain.embeddings import DashScopeEmbeddings

# 加载知识库
index = FAISS.load_local("./knowledge_base/faiss_index")
```

## 七、常见问题

### Q1: 构建失败，提示 API Key 错误

**解决**：
```bash
# 检查 API Key 是否正确设置
echo $DASHSCOPE_API_KEY  # Mac/Linux
echo %DASHSCOPE_API_KEY%  # Windows

# 重新设置
export DASHSCOPE_API_KEY="sk-xxxxx"
```

### Q2: PDF 解析出乱码

**原因**：扫描版 PDF 或编码问题

**解决**：
- 优先使用 Word 格式
- PDF 转为可编辑版本（OCR 识别）

### Q3: 知识库太大，加载慢

**优化**：
- 减少片段大小（chunk_size 从 500 改为 300）
- 过滤低质量文档
- 分类构建多个索引
